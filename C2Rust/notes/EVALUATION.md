# C to Safe Rust Translation: Evaluation Methodology

> 本文档综合分析C-to-Safe-Rust领域的评估方法论，包括使用的基准测试集和关键评估指标。

---

## 一、评估基准 (Benchmarks)

### 1. CRUST-Bench (UT Austin, 2025)

**特点**: 首个专门为C-to-Safe-Rust设计的标准化基准

| 属性 | 值 |
|------|-----|
| 规模 | 100个C语言仓库 |
| 平均代码量 | 958 LoC/仓库 |
| 测试覆盖率 | 67% |
| 开源 | https://github.com/AnirudhKhatry/CRUST-bench |

**创新设计**:
- **手写Rust接口**: 人工标注目标Rust函数签名和类型定义
- **功能测试**: 为每个仓库编写Rust单元测试
- **标准化流程**: 输入C代码+接口 → 生成实现 → 编译 → 测试

**适用场景**: LLM代码翻译能力评估，Safe Rust生成质量评估

---

### 2. Laertes Corpus (UCSB, 2021)

**特点**: 学术界最早使用的C-to-Rust评估程序集

| 属性 | 值 |
|------|-----|
| 规模 | 17个C程序 |
| 程序类型 | 真实开源项目 |
| 开源 | 随Laertes论文发布 |

**包含程序**:
- 压缩工具: bzip2, gzip
- JSON解析: json-c
- 机器学习: genann
- 其他: urlparser, libcsv, grabc, lil, RoaringBitmap, optfuzz等

**适用场景**: unsafe消除分析，指针转换率评估

---

### 3. Rosetta Code

**特点**: 社区维护的多语言算法实现集

| 属性 | 值 |
|------|-----|
| 规模 | 126+个程序（C部分） |
| 程序类型 | 算法实现，函数级 |
| 来源 | https://rosettacode.org |

**特点**:
- 每个程序相对独立
- 多有标准输入输出
- 适合函数级翻译评估

**适用场景**: 函数级翻译，快速原型验证

## 二、核心评估指标 (Metrics)

### 1. 编译相关指标

#### 1.1 编译成功率 (Compilation Success Rate)
```
编译成功率 = 编译通过的翻译数 / 总翻译数
```
- **意义**: 最基本的正确性指标
- **局限**: 编译通过≠功能正确

#### 1.2 行接受率 (Line Acceptance Rate)
```
行接受率 = 被Rust编译器接受的代码行数 / 总翻译代码行数
```
- **提出者**: EvoC2Rust
- **意义**: 更细粒度的编译质量评估

---

### 2. 功能正确性指标

#### 2.1 测试通过率 (Test Pass Rate)
```
测试通过率 = 通过测试的翻译数 / 总翻译数
```
- **意义**: 验证功能等价性
- **依赖**: 需要测试用例

#### 2.2 断言通过率 (Assertion Pass Rate)
```
断言通过率 = 通过的断言数 / 总断言数
```
- **提出者**: Rustine
- **意义**: 比测试通过率更细粒度

#### 2.3 等价性验证
- **Syzygy**: 比较C和Rust版本的I/O行为
- **FLOURINE**: 使用fuzz testing验证

---

### 3. 安全性指标

#### 3.1 Unsafe代码比例
```
Unsafe比例 = unsafe代码行数 / 总代码行数
```

#### 3.2 原始指针统计
| 指标 | 定义 |
|------|------|
| Raw Pointer声明数 | `*mut T` / `*const T` 的声明数量 |
| Raw Pointer解引用数 | `*ptr` 操作的数量 |
| Unsafe类型转换数 | `as` 关键字用于不安全转换的数量 |

#### 3.3 Unsafe分类 (Laertes定义)

| 类别 | 说明 | 典型来源 |
|------|------|----------|
| RawDeref | 原始指针解引用 | `*ptr` |
| Global | 全局可变状态 | `static mut` |
| Union | 联合类型访问 | `union.field` |
| Allocation | 不安全内存分配 | 自定义allocator |
| Extern | 外部函数调用 | FFI |
| Cast | 不安全类型转换 | `transmute` |
| InlineAsm | 内联汇编 | `asm!` |

#### 3.4 指针转换率
```
指针转换率 = 转换为引用的指针数 / 总原始指针数
```
- **Laertes**: 整体~11%，合格指针~93%
- **Aliasing Limits**: 改进至~21%

---

### 4. 代码质量指标

#### 4.1 惯用性 (Idiomaticity)
- **Clippy警告数**: Rust官方lint工具的警告
- **惯用模式使用**: 如`Option`、`Result`、迭代器等

#### 4.2 可读性
- **命名规范**: 遵循Rust命名约定
- **代码结构**: 模块化、函数长度等

#### 4.3 性能
- **运行时间比较**: Rust版本 vs C版本
- **内存使用比较**: 通常Rust略高（因安全检查）

---

### 5. 效率指标

#### 5.1 成本
```
API成本 = LLM API调用费用
```
| 工具 | 报告成本 |
|------|----------|
| Syzygy | ~$2500 |
| Rustine | ~$0.48 |
| 其他 | 未报告 |

#### 5.2 时间
- 翻译时间
- 编译时间
- 测试时间

## 四、CRUST-Bench基准结果

### LLM性能对比 (Pass@1)

| 模型 | 编译通过率 | 测试通过率 |
|------|------------|------------|
| OpenAI o1 | 47% | **15%** |
| Claude 3.5 Sonnet | 43% | 12% |
| GPT-4o | 39% | 10% |
| DeepSeek v3 | 36% | 9% |
| GPT-4o-mini | 28% | 6% |

### 修复后提升

| 模型 | 原始 | 修复后 | 提升 |
|------|------|--------|------|
| o1 | 15% | 37% | +22% |
| Claude 3.5 | 12% | 28% | +16% |
| GPT-4o | 10% | 25% | +15% |

**关键结论**: 即使最强LLM也仅15%测试通过率，说明问题远未解决。

## 五、Laertes语料库Unsafe分布

### 按类别统计

| Unsafe类别 | 占比 | 可消除性 |
|------------|------|----------|
| RawDeref | ~95% | 部分可消除 |
| Global | ~2% | 难以消除 |
| Extern | ~2% | 无法消除 |
| Cast | <1% | 部分可消除 |
| Union | <1% | 难以消除 |
| Allocation | <1% | 难以消除 |
| InlineAsm | <0.1% | 无法消除 |

### 指针转换结果

| 方法 | 总指针转换率 | 合格指针转换率 |
|------|--------------|----------------|
| Laertes基线 | 11% | 93% |
| +子集分析 | 16% | - |
| +上下文敏感 | 19% | - |
| +伪安全 | 21% | - |

**关键发现**: 89%的指针因别名等原因根本无法转换。

## 六、评估建议

### 最小评估集
对于新C-to-Rust工作，建议至少使用：
1. **CRUST-Bench**: 功能正确性的标准基准
2. **编译成功率 + 测试通过率**: 基本正确性
3. **Unsafe代码比例**: 安全性

### 全面评估集
```
基准: CRUST-Bench + Laertes Corpus + 自选大项目

指标:
├── 正确性
│   ├── 编译成功率
│   ├── 测试通过率
│   └── 断言通过率（如适用）
├── 安全性
│   ├── Unsafe代码比例
│   ├── Raw Pointer统计
│   └── 指针转换率
├── 质量
│   ├── Clippy警告数
│   └── 性能对比
└── 效率
    ├── API成本
    └── 翻译时间
```

## 七、当前评估的局限

1. **规模有限**: 最大评估规模~91K LoC，真实工业项目更大
2. **测试依赖**: 功能验证依赖测试，测试可能不完整
3. **人工因素**: 部分基准（如CRUST-Bench）有人工标注，可能引入偏见
4. **单一语言**: 仅评估C到Rust，未考虑C++等更复杂情况
5. **忽略性能**: 大多数工作不评估性能等价性
